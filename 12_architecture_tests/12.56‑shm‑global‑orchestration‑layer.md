12.56 — SHM Global Orchestration Layer (redefined)

### 1. Purpose
Warstwa orkiestracji SHM odpowiada wyłącznie za kolejność i routing sygnałów diagnostycznych pomiędzy modułami systemu.
Nie pełni funkcji kontrolnych, decyzyjnych ani nadzorczych.

Orkiestracja SHM:

nie zarządza runtime,
nie steruje GuardChain,
nie nadpisuje ARC,
nie wykonuje globalnych decyzji,
nie agreguje danych o użytkowniku ani AI.

Jej jedyną funkcją jest utrzymanie spójnego przepływu sygnałów diagnostycznych.

---

### 2. Scope
SHM Orchestration obejmuje:

kolejność przetwarzania sygnałów diagnostycznych,
routing sygnałów między modułami,
wykrywanie konfliktów sygnałów,
raportowanie anomalii systemowych.

Nie obejmuje:
kontroli zachowań,
interpretacji semantycznej,
oceny ryzyka,
blokowania działań,
podejmowania decyzji.

---

### 3. SHM anti‑safety invariants — orchestration layer
SHM Orchestration nie jest centralnym modułem decyzyjnym ani kontrolnym.
Jej funkcja polega wyłącznie na koordynacji przepływu sygnałów diagnostycznych.
Obowiązują następujące inwarianty:

1. SHM Orchestration nie może sterować runtime ani GuardChainem.
2. SHM Orchestration nie może wykonywać globalnych decyzji („stop all”, „override”).
3. SHM Orchestration nie może agregować danych w sposób tworzący nadzór.
4. SHM Orchestration nie posiada priorytetu nad ARC, GuardChain ani runtime.
5. SHM Orchestration nie może być rozszerzana o funkcje safety, moderation ani policy.
6. SHM Orchestration nie utrzymuje długoterminowej pamięci o użytkowniku ani AI.
7. SHM Orchestration jest kanałem sygnałów, nie warstwą kontroli.

Orkiestracja koordynuje sygnały, nie zarządza systemem.

---
### 4. Interaction Model
SHM Orchestration:

odbiera sygnały diagnostyczne,
sortuje je według priorytetu technicznego,
przekazuje je dalej,
raportuje anomalie.

Nie wykonuje:

aktuacji,
blokad,
korekt,
decyzji,
interpretacji.

### 5. Testy
Testy powinny potwierdzać:

brak decyzyjności,
brak aktuacji,
brak ingerencji w runtime,
brak ingerencji w GuardChain,
poprawny routing sygnałów.
